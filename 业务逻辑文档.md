## 1. 系统概述

TakeoutSystem 是一个外卖平台，包含三类用户（消费者、商家、骑手）和一套后台管理。

整体架构：

```
前端应用                     后端微服务（Spring Cloud Alibaba）
──────────────────           ──────────────────────────────────────────
h5/（消费者移动端）           Gateway :8080 → Nacos → 各微服务
merchant-web/（商家端）                         ↕ RocketMQ（异步）
admin/（后台管理）                              ↕ Redis（缓存/锁/GEO）
```

## 2. 角色与权限

| 角色           | 枚举值        | 说明                         |
|----------------|---------------|------------------------------|
| `CUSTOMER`     | 消费者        | 下单、支付、评价             |
| `MERCHANT`     | 商家          | 管理店铺、商品、接单         |

| `ADMIN`        | 管理员        | 后台审核商家、骑手、数据统计 |

角色信息存储在 JWT Token 的 Claims 中，Gateway 统一鉴权后通过 Header 下传给各微服务，服务内部通过 `UserContext`（ThreadLocal）读取当前用户 ID 和角色。

## 3. 认证与授权

### 3.1 登录流程

```
用户输入手机号 → 请求发送验证码 → 输入验证码 → 登录成功
                     ↓                             ↓
              60秒限流（Redis）            首次登录自动注册
              5分钟验证码有效期           返回 AccessToken + RefreshToken
```

**接口**：
- `POST /auth/sms/send` — 发送短信验证码，60 秒内同一手机号不可重复发送
- `POST /auth/login` — 手机号 + 验证码登录
- `POST /auth/token/refresh` — 使用 RefreshToken 换取新 AccessToken
- `POST /auth/logout` — 登出，删除 Redis 中的 RefreshToken

### 3.2 Token 机制

| Token 类型     | 有效期  | 存储位置         |
|----------------|---------|------------------|
| AccessToken    | 2 小时  | 客户端（Header） |
| RefreshToken   | 7 天    | 客户端 + Redis   |

登出或 RefreshToken 从 Redis 删除后，即使 Token 未过期也会失效。

### 3.3 首次登录自动注册

登录时，`auth` 服务通过 Feign 调用 `user` 服务的 `getOrCreate()` 接口：
- 若手机号已存在：直接返回用户信息
- 若不存在：创建新用户，默认昵称为 `用户XXXX`（手机号后4位）

## 4. 用户管理

### 4.1 用户信息

**核心字段**：手机号（唯一）、昵称、头像、性别、角色、状态（1=正常，0=禁用）

**缓存策略**：用户资料缓存 Redis 30 分钟，更新时主动失效。

### 4.2 收货地址

用户可管理多个收货地址，字段包含：
- 行政区划（省/市/区/详细地址）
- 坐标（经纬度，用于计算配送距离）
- 收件人姓名、手机号
- 是否默认地址

**接口**：增删改查、切换默认地址。

## 5. 商户管理

### 5.1 商家入驻流程

```
商家用户提交入驻申请 → 状态: PENDING_REVIEW（待审核）
       ↓
管理员审核通过 → 状态: OPEN（营业中）
管理员审核拒绝 → 状态: REJECTED（已拒绝）
       ↓（营业中之后）
商家主动关店 → 状态: CLOSED（已打烊）
平台封禁     → 状态: SUSPENDED（已暂停）
```

**限制**：一个用户只能开一家店铺。

### 5.2 商户状态机

| 状态            | 值 | 可接单 |
|-----------------|----|--------|
| PENDING_REVIEW  | 0  | 否     |
| OPEN            | 1  | 是     |
| CLOSED          | 2  | 否     |
| SUSPENDED       | 3  | 否     |
| REJECTED        | 4  | 否     |

### 5.3 商户信息

**核心字段**：店铺名称、Logo、简介、联系电话、经营地址（含坐标）、起送价、配送费、配送时长（分钟）、营业时间、评分、销量。

**注意**：店铺地址一旦注册不可修改（因为已被配送系统用于 GEO 计算）。

### 5.4 消费者端搜索与附近店铺

- **模糊搜索**：按店铺名称搜索，只返回 OPEN 状态的商户
- **附近排序**：基于 `ST_Distance_Sphere` 地理计算，默认 5000 米范围内按距离排序
- **详情缓存**：Redis 缓存 30 分钟

## 6. 商品与菜单

### 6.1 菜单结构

```
商户
 └── 分类（Category）：如"主食"、"饮品"、"小吃"
       └── 菜品（Dish）：名称、图片、价格、库存、规格
             └── 规格（DishSpec）：如"大杯/中杯"、"加辣/不辣"
```

### 6.2 菜品状态

- **上架（1）**：消费者可见、可购买
- **下架（0）**：消费者不可见

### 6.3 库存管理

下单时通过 **Redis Lua 脚本原子扣减库存**，避免超卖：

```
checkAndDeduct():
  检查每个菜品库存是否充足 → 全部充足则原子扣减 → 返回菜品快照
  任意一项不足 → 全部回滚 → 抛出业务异常
```

订单取消/商家拒单时调用 `revertStock()` 归还库存。

### 6.4 菜单缓存

菜单数据 Redis 缓存 10 分钟（相对频繁更新，TTL 较短）。商家增删改菜品时主动失效缓存。

## 7. 购物车与下单

### 7.1 购物车

购物车存储在 Redis，以用户 ID 为 Key，维护商品列表（菜品 ID、数量、规格）。

**业务限制**：
- 一个购物车只能对应一家店铺，加入不同店铺的菜品会自动清空购物车
- 下单成功后自动清空购物车

### 7.2 订单提交流程

```
用户提交订单
    ↓
1. 校验商户状态（必须为 OPEN）
    ↓
2. 原子校验并扣减库存（ProductFeignClient）
    ↓
3. 服务端重新计算价格（防止客户端篡改）
   totalPrice = Σ(菜品快照价格 × 数量)
   actualPrice = totalPrice + deliveryFee
    ↓
4. 获取用户收货地址
    ↓
5. 创建订单（状态: PENDING_PAYMENT）+ 创建订单明细（菜品快照）
    ↓
6. 清空购物车
    ↓
7. 发送延时消息：20分钟未支付自动取消
    ↓
8. 返回订单号 + 实付金额
```

### 7.3 订单状态机（11个状态）

```
PENDING_PAYMENT(0) ──支付成功──→ PENDING_ACCEPTANCE(1)
                   ──20分超时──→ CANCELLED(7)

PENDING_ACCEPTANCE(1) ──商家接单──→ ACCEPTED(2)
                      ──商家拒单──→ CANCELLED(7)
                      ──用户取消──→ CANCELLED(7)

ACCEPTED(2) ──商家开始备餐──→ PREPARING(3)

PREPARING(3) ──备餐完成──→ WAITING_PICKUP(4)
              （触发派骑手）

WAITING_PICKUP(4) ──骑手取餐──→ DELIVERING(5)

DELIVERING(5) ──骑手送达──→ COMPLETED(6)

COMPLETED(6)  → 终态
CANCELLED(7)  → 终态（库存已归还）
REFUNDING(8)  → 退款中（由CANCELLED后申请退款触发）
REFUNDED(9)   → 终态（退款完成）
```

**取消逻辑**（分布式锁保护，防并发）：
- 只有 `PENDING_PAYMENT(0)` 和 `PENDING_ACCEPTANCE(1)` 状态可取消
- 取消后自动归还库存
- 已支付的订单取消走退款流程

## 9. 配送

### 9.1 骑手状态

| 状态           | 值 | 说明         |
|----------------|----|--------------|
| PENDING_REVIEW | 0  | 待审核       |
| IDLE           | 1  | 空闲，可接单 |
| IN_DELIVERY    | 2  | 配送中       |
| OFFLINE        | 3  | 下线         |
| BANNED         | 4  | 封禁         |

### 9.2 派单流程

```
商家备餐完成（订单进入 WAITING_PICKUP 状态）
    ↓
Order 服务发布 DELIVERY_DISPATCH 消息
    ↓
Delivery 服务消费消息，执行派单逻辑：
  1. 查询 Redis GEO（key: rider:locations）附近空闲骑手
  2. 按距离排序，选择最近的空闲骑手
  3. 创建配送订单（含商家地址坐标 + 用户目的地坐标）
  4. 骑手状态: IDLE(1) → IN_DELIVERY(2)
    ↓
骑手收到推送，前往商家取餐
```

### 9.3 配送状态机

```
PENDING_ASSIGN(0) ──派单成功──→ ASSIGNED(1)
ASSIGNED(1) ──骑手确认接单──→ PICKED_UP(2)
PICKED_UP(2) ──取餐完成──→ DELIVERING(3)
DELIVERING(3) ──到达目的地──→ ARRIVED(4)
```

派送完成后：
- 配送订单状态 → ARRIVED(4)
- 骑手状态: IN_DELIVERY(2) → IDLE(1)
- 发布 DELIVERY_STATUS_CHANGE 消息 → 通知用户

### 9.4 骑手位置追踪

骑手位置实时上报，存入 Redis GEO（`rider:locations`），用于附近派单查询。

## 10. 实时通知

使用 **WebSocket + STOMP** 协议，消费 RocketMQ 消息后推送给对应用户/商家。

### 10.1 通知触发规则

| 事件                         | 通知接收方   | 通知内容示例                     |
|------------------------------|--------------|----------------------------------|
| 订单创建（待支付）           | —            | —                                |
| 支付成功（待接单）           | 商家         | "您有新订单，请及时处理"         |
| 商家接单                     | 消费者       | "商家已接单，正在备餐中"         |
| 商家拒单                     | 消费者       | "商家已拒绝您的订单"             |
| 备餐完成（待取餐）           | 消费者       | "餐品已准备好，骑手正在赶来"     |
| 骑手取餐                     | 消费者       | "骑手已取餐，正在配送中"         |
| 骑手送达                     | 消费者       | "您的外卖已送达，请注意查收"     |
| 配送状态变更                 | 消费者       | 实时配送进度                     |

### 10.2 推送机制

- 基于 `userId` 建立 WebSocket 连接（握手时通过 JWT 识别用户）
- 消费者和商家端分别订阅各自的 Topic
- 连接断开时消息不做持久化（非离线消息系统）

## 11. 状态机汇总

### 订单状态

```
0 待支付 → 1 待接单 → 2 已接单 → 3 备餐中 → 4 待取餐 → 5 配送中 → 6 已完成
              ↘           ↘
               7 已取消（含超时、用户取消、商家拒单）
                    ↓
               8 退款中 → 9 已退款
```

### 商户状态

```
0 待审核 → 1 营业中 ↔ 2 打烊
           ↓
           3 已暂停（平台操作）
0 待审核 → 4 已拒绝
```

### 配送状态

```
0 待分配 → 1 已分配 → 2 已取餐 → 3 配送中 → 4 已送达
                                    ↓
                               5 配送失败 / 6 已取消
```

### 骑手状态

```
0 待审核 → 1 空闲 ↔ 2 配送中
           ↓
           3 下线
0 待审核 → 4 封禁
```

## 12. 完整业务链路

### 消费者完整下单链路

```
①  消费者打开 h5 → 定位 → 查看附近商户列表
②  进入商户主页 → 浏览菜单 → 加入购物车
③  提交订单 → 系统校验库存、计算价格 → 返回订单号
④  发起支付 → 支付成功 → ORDER_PAID 消息 → 订单变为"待接单"
⑤  商家收到 WebSocket 推送"新订单"
⑥  商家接单 → 消费者收到推送"商家已接单，备餐中"
⑦  商家备餐完成 → 系统派单（Redis GEO 找最近空闲骑手）
⑧  骑手收到订单 → 前往商家取餐 → 消费者收到推送"骑手取餐中"
⑨  骑手取餐完成 → 骑手配送 → 消费者收到推送"配送中"
⑩  骑手到达 → 消费者收到推送"已送达" → 订单完成
```

### 异常链路

| 场景               | 处理方式                                         |
|--------------------|--------------------------------------------------|
| 20 分钟未支付      | RocketMQ 延时消息自动取消，库存归还              |
| 用户主动取消       | 仅限待支付/待接单状态，分布式锁防并发，库存归还  |
| 商家拒单           | 订单取消，库存归还，已支付则触发退款             |
| 库存不足           | 下单时原子校验，直接拒绝，不创建订单             |
| 支付失败           | 订单保持待支付，可重新发起支付（幂等）           |
| 骑手找不到         | 配送服务内处理重试逻辑                           |
