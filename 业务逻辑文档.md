## 第一章：系统概述

### 1.1 项目背景与定位

TakeoutSystem 是一个**外卖平台单体应用**，由早期微服务架构改造合并而来。平台服务三类角色：

- **消费者（CUSTOMER）**：浏览附近商户、点餐下单、支付、评价、收藏
- **商家（MERCHANT）**：管理店铺信息、菜品分类、接收并处理订单
- **管理员（ADMIN）**：审核商家入驻申请、管理用户账号、查看全平台订单

### 1.5 核心设计原则

- **统一响应格式**：所有接口返回 `Result<T>`，含 code / message / data
- **软删除**：所有用户数据表含 `deleted` 字段（0=正常，1=已删除），物理数据不丢失
- **雪花 ID**：所有主键用 Snowflake 生成，序列化为 String 防止 JS 精度丢失
- **Redis 分布式锁**：订单状态更新、优惠券领取等并发场景均用 Redis setIfAbsent 加锁
- **Lua 脚本原子操作**：库存扣减使用 Redis Lua 脚本，保证多菜品原子扣减
- **ThreadLocal 用户上下文**：拦截器解析 JWT 后存入 UserContext，请求结束后清理

## 第三章：数据库设计

### 3.1 概述

项目使用**单一数据库 `db_takeout`**，所有业务表集中在一个库中（由原微服务的多库合并而来）。

字符集：`utf8mb4`，排序规则：`utf8mb4_unicode_ci`。

MyBatis-Plus 全局配置软删除：`deleted=0` 为正常，`deleted=1` 为已删除，所有查询自动加 `WHERE deleted=0` 过滤。

### 3.2 t_user — 用户表

```sql
CREATE TABLE t_user (
  id            BIGINT      NOT NULL PRIMARY KEY COMMENT '雪花算法主键',
  phone         VARCHAR(20) NOT NULL UNIQUE       COMMENT '手机号（登录凭证，唯一）',
  nickname      VARCHAR(50)                       COMMENT '昵称（默认"用户"+手机后4位）',
  avatar_url    VARCHAR(500)                      COMMENT '头像 URL',
  gender        TINYINT     DEFAULT 0             COMMENT '性别：0=未知，1=男，2=女，3=保密',
  role          VARCHAR(20) NOT NULL DEFAULT 'CUSTOMER' COMMENT '角色：CUSTOMER/MERCHANT/RIDER/ADMIN',
  status        TINYINT     DEFAULT 1             COMMENT '状态：0=禁用，1=正常',
  created_at    DATETIME                          COMMENT '注册时间（自动填充）',
  updated_at    DATETIME                          COMMENT '最后更新时间（自动填充）',
  deleted       TINYINT     DEFAULT 0             COMMENT '软删除：0=正常，1=已删除'
);
```

**索引**：

- `UNIQUE (phone)` — 手机号唯一，登录时快速定位

**注意**：`id` 为 BIGINT 但在 Java 中用 Long 存储，JSON 序列化时转为 String 输出（防 JS 精度丢失）。

### 3.3 t_user_address — 用户收货地址表

```sql
CREATE TABLE t_user_address (
  id          BIGINT       NOT NULL PRIMARY KEY,
  user_id     BIGINT       NOT NULL               COMMENT '所属用户 ID',
  receiver    VARCHAR(50)  NOT NULL               COMMENT '收件人姓名',
  phone       VARCHAR(20)  NOT NULL               COMMENT '收件人手机号',
  province    VARCHAR(50)                         COMMENT '省',
  city        VARCHAR(50)                         COMMENT '市',
  district    VARCHAR(50)                         COMMENT '区/县',
  detail      VARCHAR(200)                        COMMENT '详细地址',
  longitude   DECIMAL(10,7)                       COMMENT '经度（用于配送距离计算）',
  latitude    DECIMAL(10,7)                       COMMENT '纬度',
  is_default  TINYINT      DEFAULT 0              COMMENT '是否默认地址：0=否，1=是',
  created_at  DATETIME,
  updated_at  DATETIME,
  deleted     TINYINT      DEFAULT 0
);
```

**索引**：

### 4.1 统一响应 Result\<T\>

```java
// com.takeout.common.result.Result<T>
public class Result<T> {
    private Integer code;    // 状态码（见 ResultCode）
    private String message;  // 提示信息
    private T data;          // 业务数据（可为 null）

    public static <T> Result<T> success(T data)   // 200 + data
    public static <T> Result<T> success()          // 200 无 data
    public static <T> Result<T> fail(ResultCode resultCode, String message)
    public static <T> Result<T> fail(String message) // 500
}
```

所有 Controller 方法返回 `Result<?>` 或 `Result<PageResult<?>>`，前端通过 `code === 200` 判断成功。

### 4.3 错误码枚举 ResultCode

| 枚举值           | code | HTTP 含义  | 使用场景                        |
| ---------------- | ---- | ---------- | ------------------------------- |
| `SUCCESS`        | 200  | 成功       | 所有正常响应                    |
| `FAIL`           | 500  | 服务器错误 | 未预期异常                      |
| `UNAUTHORIZED`   | 401  | 未认证     | Token 缺失/过期                 |
| `FORBIDDEN`      | 403  | 无权限     | 角色不匹配/操作越权             |
| `NOT_FOUND`      | 404  | 资源不存在 | 查询不到指定记录                |
| `PARAM_ERROR`    | 400  | 参数错误   | 校验失败（@Valid）              |
| `BUSINESS_ERROR` | 600  | 业务异常   | 库存不足/状态不对等业务规则违反 |

### 4.4 业务异常 BusinessException

```java
// com.takeout.common.exception.BusinessException
public class BusinessException extends RuntimeException {
    private final ResultCode resultCode;
    private final String message;

    // 常用构造方式：
    throw new BusinessException(ResultCode.NOT_FOUND, "商户不存在");
    throw new BusinessException(ResultCode.BUSINESS_ERROR, "库存不足");
    throw new BusinessException(ResultCode.FORBIDDEN, "无权操作此订单");
}
```

**重要约定**：所有业务错误必须抛 `BusinessException`，禁止抛 `IllegalStateException` 或裸 `RuntimeException`——`GlobalExceptionHandler` 只捕 `BusinessException`，其他异常会返回 500 并丢失 ResultCode 信息。

### 4.5 全局异常处理 GlobalExceptionHandler

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 1. 业务异常 → 返回对应 ResultCode
    @ExceptionHandler(BusinessException.class)
    public Result<?> handleBusinessException(BusinessException e)
    // log.warn 级别记录，不报警

    // 2. @Valid 参数校验失败
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<?> handleValidException(MethodArgumentNotValidException e)
    // → PARAM_ERROR(400)，返回第一个字段的校验错误信息

    // 3. @Validated 约束违反
    @ExceptionHandler(ConstraintViolationException.class)
    public Result<?> handleConstraintException(ConstraintViolationException e)
    // → PARAM_ERROR(400)

    // 4. 其他未预期异常
    @ExceptionHandler(Exception.class)
    public Result<?> handleException(Exception e)
    // log.error 级别，返回 FAIL(500) + 通用提示
}
```

### 4.9 雪花 ID SnowflakeIdUtil

自实现 Snowflake 算法，64 位 Long 结构：


- 理论 QPS：4096/ms（每毫秒 4096 个不重复 ID）
- `SnowflakeIdUtil.nextId()` — 返回 Long
- `SnowflakeIdUtil.nextIdStr()` — 返回 String（用于 orderNo 等需要字符串的场景）

### 5.7 AuthInterceptor

```java
@Component
public class AuthInterceptor implements HandlerInterceptor {

    // 白名单（这些路径不需要 Token）
    private static final List<String> WHITE_LIST = Arrays.asList(
        "/api/auth/**",           // 登录/注册/发送验证码
        "/api/health",            // 健康检查
        "/api/merchant/nearby",   // 附近商户（公开）
        "/api/merchant/search",   // 搜索商户（公开）
        "/api/merchant/*/detail", // 商户详情（公开）
        "/api/product/menu/**",   // 菜单（公开）
        "/api/review/merchant/**",// 商户评价（公开）
        "/doc.html",              // Swagger UI
        "/v3/api-docs/**",        // OpenAPI JSON
        "/swagger-ui/**"
    );

    @Override
    public boolean preHandle(HttpServletRequest request, ...) {
        // 1. 取 Authorization 头
        String token = request.getHeader("Authorization");
        if (token == null || !token.startsWith("Bearer ")) {
            throw new BusinessException(ResultCode.UNAUTHORIZED, "未登录");
        }
        // 2. 解析 JWT
        Claims claims = JwtUtil.parseToken(token.substring(7), jwtProperties.getSecret());
        // 3. 存入 UserContext
        UserContext.setUserId(Long.parseLong(claims.getSubject()));
        UserContext.setUserRole(claims.get("role", String.class));
        return true;
    }

    @Override
    public void afterCompletion(...) {
        UserContext.clear();  // 必须清理，防 ThreadLocal 内存泄漏
    }
}
```

## 第六章：认证模块

### 6.1 模块概述

认证模块负责：手机号 + 短信验证码登录、JWT Token 颁发与刷新、登出。首次登录自动注册账号（无单独注册接口）。

**包路径**：`com.takeout.auth`

### 6.2 发送短信验证码

**接口**：`POST /api/auth/sms/send`  
**请求体**：`SmsCodeRequest { phone: "13800000001" }`  
**无需认证**

流程：

```
1. 参数校验（手机号格式正则）
2. 检查限流 key: sms:limit:{phone}（Redis）
   ├─ 存在 → 抛 BusinessException("验证码发送过于频繁，请60秒后重试")
   └─ 不存在 → 继续
3. 生成 6 位验证码
   开发环境：固定 "123456"
   生产环境：调用短信服务（当前为模拟）
4. 存入 Redis: sms:code:{phone} = "123456"，TTL = 5 分钟
5. 存入限流 key: sms:limit:{phone} = "1"，TTL = 60 秒
6. 返回 Result.success()
```

### 6.3 登录

**接口**：`POST /api/auth/login`  
**请求体**：`LoginRequest { phone, code }`  
**无需认证**

流程：

```
1. 参数校验（手机号格式、code 不为空）
2. 查询 Redis: sms:code:{phone}
   ├─ 不存在 → "验证码已过期，请重新发送"
   └─ 存在 → 比对 code
       ├─ 不匹配 → "验证码错误"
       └─ 匹配 → 继续
3. 删除验证码 key（一次性使用）
4. getOrCreate(phone)：
   ├─ 查 t_user WHERE phone=? AND deleted=0
   ├─ 存在 → 返回已有用户
   └─ 不存在 → INSERT：nickname="用户"+phone后4位, role=CUSTOMER, status=1
5. 检查用户状态：status=0 → "账号已被禁用"
6. 生成 accessToken（携带 userId + role，2小时有效）
7. 生成 refreshToken（仅携带 userId，7天有效）
8. 存 refreshToken: user:token:{userId} = refreshToken，TTL = 7天
9. 返回 LoginVO { accessToken, refreshToken, userInfo }
```

**LoginVO 结构**：

```json
{
  "accessToken": "eyJ...",
  "refreshToken": "eyJ...",
  "userInfo": {
    "id": "1234567890123456789",
    "phone": "13800000001",
    "nickname": "用户0001",
    "avatarUrl": null,
    "role": "CUSTOMER"
  }
}
```

### 6.4 Token 刷新

**接口**：`POST /api/auth/refresh`  
**请求体**：`RefreshTokenRequest { refreshToken }`  
**无需认证**

流程：

```
1. 解析 refreshToken（验证签名和过期时间）
   └─ 异常 → "Token 无效或已过期"
2. 取 userId from claims
3. 查 Redis: user:token:{userId}
   ├─ 不存在（已登出或过期） → "登录已过期，请重新登录"
   └─ 存在且与请求 token 一致 → 继续
4. 查用户信息（验证账号未被禁用）
5. 生成新 accessToken
6. 返回新 accessToken（refreshToken 不变）
```

### 6.5 登出

**接口**：`POST /api/auth/logout`  
**需要认证**（从 UserContext 取 userId）

```
1. 取 UserContext.getUserId()
2. 删除 Redis: user:token:{userId}
3. 返回 Result.success()
```

登出后原 refreshToken 失效，即使 accessToken 未过期，刷新时也会失败。

```sql
           POINT(longitude, latitude),
           POINT(#{longitude}, #{latitude})

FROM t_merchant
WHERE deleted = 0
  AND status = 1
  AND ST_Distance_Sphere(
          POINT(longitude, latitude),
          POINT(#{longitude}, #{latitude})
      ) <= #{radius}
ORDER BY distance ASC
LIMIT #{offset}, #{size}
```

### 9.4 库存管理（Lua 脚本原子扣减）

下单时调用 `DishService.checkAndDeduct(List<StockDeductItem>)`:

```lua
-- Redis Lua 脚本（STOCK_DEDUCT_LUA）
-- KEYS: [dish:stock:id1, dish:stock:id2, ...]
-- ARGV: [qty1, qty2, ...]

-- 第一阶段：校验所有菜品库存
for i = 1, #KEYS do
    local stock = redis.call('GET', KEYS[i])
    if stock == nil then
        -- Redis 中无 key：-1（不限量），允许通过
    elseif tonumber(stock) < tonumber(ARGV[i]) then
        return -i   -- 返回第 i 个菜品库存不足
    end
end

-- 第二阶段：原子扣减（全部满足才执行）
for i = 1, #KEYS do
    local stock = redis.call('GET', KEYS[i])
    if stock ~= nil then
        redis.call('DECRBY', KEYS[i], tonumber(ARGV[i]))
    end
end
return 0  -- 扣减成功
```

**Redis 库存 key 初始化时机**：

- 新增菜品时（stock >= 0）：`SET dish:stock:{id} = stock`
- 修改菜品 stock 字段时：同步更新 Redis
- key 不存在（stock=-1）：跳过扣减，视为不限量

**库存归还**（订单取消/拒单时调用）：

```java
// DishService.revertStock(List<StockDeductItem>)
for (StockDeductItem item : items) {
    String key = "dish:stock:" + item.dishId();
    if (redisUtil.hasKey(key)) {
        redisUtil.increment(key, item.quantity()); // INCRBY
    }
    // DB 层：UPDATE t_dish SET stock=stock+? WHERE id=?
}
```

同时更新 DB 中的 `t_dish.stock`，保持 Redis 与 DB 最终一致。

### 11.2 提交订单完整流程

**POST /api/order/submit** — 需要认证（CUSTOMER）

`SubmitOrderRequest { merchantId, addressId, items(List<CartItemDTO>), remark, payType, userCouponId }`

```
@Transactional(rollbackFor = Exception.class)

1. 校验商户状态
   └─ status != 1 → "商户暂未营业"

2. 构建扣减列表
   将 items 中相同 dishId 的合并（防客户端重复传）

3. 原子扣减库存（Lua 脚本）
   └─ 返回 -N → 第 N 个菜品库存不足，抛 BusinessException
   └─ 返回 0  → 扣减成功，获得 DishSnapshot 列表

4. 服务端重新计算价格（防篡改）
   totalPrice = Σ(snapshot.unitPrice × quantity)
   actualPrice = totalPrice + merchant.deliveryFee

5. 处理优惠券（若 userCouponId 非 null）
   a. 查 t_user_coupon WHERE id=? AND user_id=? AND status=0
      不存在 → "优惠券不存在或已使用"
   b. 检查优惠券有效期
   c. 检查起用门槛：totalPrice < minOrderPrice → "未满足使用门槛"
   d. discount = coupon.discount
   e. actualPrice = actualPrice - discount（最低为 0）
   （此时不 markUsed，等支付成功后再标记）

6. 查收货地址
   └─ 不存在或不属于当前用户 → "收货地址不存在"

7. 生成订单号：SnowflakeIdUtil.nextIdStr()

8. INSERT t_order（status=1，快照地址信息）

9. 批量 INSERT t_order_item（快照菜品信息）

10. 清空购物车
    DELETE t_cart WHERE user_id=? AND merchant_id=?

11. 若有优惠券：标记使用
    UPDATE t_user_coupon SET status=1, used_at=now() WHERE id=?

12. 返回 SubmitOrderVO { orderNo, actualPrice }
```

**事务边界**：整个流程在一个 `@Transactional` 内，任何步骤失败均回滚。库存 Redis 扣减在事务外（Lua 脚本不支持 JDBC 事务），取消时需手动归还。

### 14.1 评价实体字段

### 14.2 提交评价

**POST /api/review** — 需要认证（CUSTOMER）

`SubmitReviewRequest { orderNo, score, content }`

```
1. 查 t_order WHERE order_no=?
   不存在或不属于当前用户 → FORBIDDEN
   status != 6 → "订单未完成，暂不支持评价"

2. 查 t_review WHERE order_no=?
   已存在 → "该订单已评价"

3. INSERT t_review {
     order_no    = orderNo,
     user_id     = currentUserId,
     merchant_id = order.merchantId,
     score       = score,
     content     = content
   }

4. 更新商户综合评分：
   avgScore = SELECT AVG(score) FROM t_review
              WHERE merchant_id=? AND deleted=0
   UPDATE t_merchant SET score=avgScore WHERE id=?

   同时清除商户 Redis 缓存（merchant:info:{merchantId}）
```

### 14.3 查询商户评价

**GET /api/review/merchant/{merchantId}** — 无需认证

```
分页查询 t_review WHERE merchant_id=? AND deleted=0
ORDER BY created_at DESC

批量加载用户昵称（enrichWithNickname）：
  收集所有 userId → 批量查 t_user → 填充 userNickname 字段
```

返回 `PageResult<ReviewVO>`。

### 14.4 查询我的评价

**GET /api/review/my** — 需要认证

```
SELECT * FROM t_review WHERE user_id=? AND deleted=0
ORDER BY created_at DESC
```

### 14.5 查询订单评价

**GET /api/review/order/{orderNo}** — 无需认证

```
SELECT * FROM t_review WHERE order_no=?
不存在 → 返回 Result.success(null)（前端据此判断是否已评价）
```

## 第十八章：缓存策略汇总

### 18.1 Redis Key 命名规范

所有 Key 使用 `:` 分隔层级，格式：`模块:类型:标识`。

| Key 模式                                  | 类型         | TTL     | 写入时机             | 失效时机                              | 用途                    |
| ----------------------------------------- | ------------ | ------- | -------------------- | ------------------------------------- | ----------------------- |
| `sms:code:{phone}`                        | String       | 5 分钟  | 发送短信时           | TTL到期 / 登录成功后主动删除          | 短信验证码              |
| `sms:limit:{phone}`                       | String       | 60 秒   | 发送短信时           | TTL到期                               | 发送频率限流            |
| `user:token:{userId}`                     | String       | 7 天    | 登录成功时           | TTL到期 / 主动登出                    | RefreshToken 有效性校验 |
| `merchant:info:{merchantId}`              | JSON Object  | 30 分钟 | 查询商户详情时       | TTL到期 / 商户信息更新/审核时主动删除 | 商户详情缓存            |
| `merchant:menu:{merchantId}`              | JSON Object  | 10 分钟 | 查询菜单时           | TTL到期 / 菜品/分类变更时主动删除     | 菜单数据缓存            |
| `dish:stock:{dishId}`                     | String (int) | 无TTL   | 新增/修改菜品stock时 | 手动删除（不常见）                    | 菜品实时库存            |
| `lock:order:{orderNo}`                    | Integer      | 30 秒   | 订单状态更新操作时   | TTL到期 / 操作完成后主动释放          | 订单状态更新分布式锁    |
| `lock:coupon:receive:{userId}:{couponId}` | Integer      | 10 秒   | 领取优惠券时         | TTL到期 / 操作完成后主动释放          | 领券分布式锁            |

### 18.2 分布式锁实现

```java
// 加锁
Boolean locked = redisUtil.setIfAbsent(lockKey, 1, ttlSeconds, TimeUnit.SECONDS);
if (!Boolean.TRUE.equals(locked)) {
    throw new BusinessException(ResultCode.BUSINESS_ERROR, "操作频繁，请稍后重试");
}

try {
    // 业务逻辑
} finally {
    redisUtil.delete(lockKey);  // 释放锁
}
```

**注意**：当前实现为简单的 `set NX EX` 模式，未使用 Lua 脚本保证原子性释放（未检查 value 一致性）。在极端情况下（业务超时 + TTL 到期 + 新持锁者），可能误删他人的锁。生产环境建议升级为 Redisson 或带 UUID value 的 Lua 释放脚本。

### 18.3 缓存与 DB 一致性策略

**Cache-Aside（旁路缓存）模式**：

```
读：先查 Redis → 命中返回 → 未命中查 DB → 写入 Redis → 返回
写：更新 DB → 主动删除 Redis（删除而非更新，避免并发写竞争）
```

**主动失效触发点**：

| 操作               | 失效的 Key                   |
| ------------------ | ---------------------------- |
| 修改商户信息       | `merchant:info:{merchantId}` |
| 审核商户           | `merchant:info:{merchantId}` |
| 修改商户状态       | `merchant:info:{merchantId}` |
| 新增/修改/删除菜品 | `merchant:menu:{merchantId}` |
| 新增/修改/删除分类 | `merchant:menu:{merchantId}` |
| 上下架菜品         | `merchant:menu:{merchantId}` |

### 19.4 Token 安全设计

| 安全点               | 实现方式                                             |
| -------------------- | ---------------------------------------------------- |
| 防 Token 伪造        | HMAC-SHA256 签名，密钥长度 44 字符                   |
| AccessToken 短有效期 | 2 小时，即使泄露危害有限                             |
| RefreshToken 可撤销  | 存 Redis，登出删除 key 即可让 Token 即时失效         |
| 防 JS 精度丢失       | Snowflake Long ID → JSON 序列化为 String             |
| 防 SQL 注入          | MyBatis-Plus 参数化查询，无拼接 SQL                  |
| 防 XSS               | 纯 API 接口，无 HTML 渲染，前端 Vue 模板自动转义     |
| CORS                 | 开发环境 allowedOriginPatterns("\*")，生产应限定域名 |

### 19.5 异常处理规范

**正确用法**：

```java
// ✅ 正确：抛 BusinessException
throw new BusinessException(ResultCode.NOT_FOUND, "商户不存在");
throw new BusinessException(ResultCode.BUSINESS_ERROR, "库存不足");
throw new BusinessException(ResultCode.FORBIDDEN, "无权操作");
```

**禁止用法**：

```java
// ❌ 错误：GlobalExceptionHandler 捕获不到，会返回 500
throw new IllegalStateException("商户不存在");
throw new RuntimeException("库存不足");
```

**GlobalExceptionHandler 捕获层级**：

```
BusinessException     → 使用 ResultCode 中的 code 和 message
@Valid 校验失败        → PARAM_ERROR(400) + 字段校验信息
ConstraintViolation   → PARAM_ERROR(400)
其他 Exception        → FAIL(500) + 通用提示（不暴露内部细节）
```

```
  │    → 获取待接单列表

  │
```

### 取消订单链路

