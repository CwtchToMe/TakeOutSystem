# 协作规则 — TakeoutSystem

> 这些规则是用户明确指定的或在真实协作中总结出来的。每次开始任务前默认遵守。

---

## 权限与确认

### 1. 所有工具权限已默认授予
用户原话：**"默认所有的权限都给你，不要再询问我了"**

- 不需要询问是否可以读文件、修改代码、运行构建命令
- 不需要询问是否可以启动/停止进程
- 直接行动，汇报结果
- **例外：** 不可逆操作（删除数据库、force push、删除分支）仍需确认

---

## 验证方式

### 2. 必须看 UI，不能只看 API
用户原话：**"你要自己去看看 UI，不要盲目看接口"**

- 接口 200 ≠ UI 正确。要用 Playwright 实际渲染页面，看用户真正看到什么
- 看到 `<!---->` (空注释节点) = 组件没渲染，不是数据问题
- 检查按钮是否存在、是否可点击，tab 切换后内容是否出现
- 执行 `/audit-ui` 做完整验证，不要只 curl 接口

### 3. 调试顺序：先看 UI 现象，再追溯根因
1. 用 Playwright 截图或检查 innerHTML
2. 看网络请求（发了什么、收到什么）
3. 对比 API 返回的真实字段名与前端绑定的字段名
4. 最后才看后端逻辑

---

## 修复策略

### 4. 发现 bug 立即修复，不只汇报
- 审计发现问题 → 直接修复 → 告知修复了什么
- 不要只列问题清单让用户自己改

### 5. 修复一个问题时，顺带检查同类问题
- 修复一个 status label 不完整 → 检查所有三端的 status label
- 修复一个 API URL 错误 → 扫描同文件中其他 API call 的 URL 是否也对
- 修复一个前端按钮缺失 → 检查同页面所有状态下是否都有对应操作

### 6. 已知坑优先排查
调试前先读 `.claude/skills/known-pitfalls/SKILL.md`，里面有 20 条历史踩坑，对号入座后再动手，节省大量时间。

---

## 响应风格

### 7. 简洁，不要摘要
- 不在每次回复末尾写"总结一下，我做了..."
- 一个任务完成 → 一两句话说做了什么、结果如何
- 代码改动靠 diff 说话，不需要散文描述

### 8. 有修改就说修改了哪个文件的哪个地方
正确：
```
修改了 merchant-web/api/index.js:14 — rejectOrder 改为用 JSON body 传 reason
```
错误：
```
我修改了相关的接口调用方式，通过将参数从查询字符串改为请求体，使后端能够正确读取...
```

### 9. 不写注释，除非 WHY 非常不明显
- 不写"这个函数用来..."类注释
- 不写"修复 bug #xxx"类注释（属于 PR 描述的内容）
- 只写隐藏约束、反直觉的不变量、特定 bug 的 workaround

---

## 技术决策偏好

### 10. 不要过度设计
- bug fix 不需要重构周围的代码
- 三处相似代码优于提前抽象
- 只解决当前的问题，不为假想的未来需求做设计

### 11. 统一使用既有模式
- 异常：`BusinessException(ResultCode.xxx, "msg")`，不造新类
- 缓存对象：`@Data` 类，不用 record
- VO：复用已有的命名约定（VO、Request、PageQuery）
